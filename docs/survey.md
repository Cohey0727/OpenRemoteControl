# open-rc — Survey of comparable projects

> **Date:** 2026-06-29
> **Purpose:** Before committing to a design, survey close analogues on GitHub to
> see if a better approach exists, and to learn from their trade-offs.
> **Result:** Approach **(b)** (stream-json subprocess, `pocket-claude` style) is
> confirmed best for our goal. See [`architecture.md`](./architecture.md) §2.

---

- "Claude Code remote control"

suggested they tackle the same problem we do (drive Claude Code from a

## 2. Detailed inspection

### 2.1 mobvibe — (c) ACP NDJSON

> "Just a open Claude Code Remote Control"
> — [Eric-Song-Nop/mobvibe](https://github.com/Eric-Song-Nop/mobvibe)

**Approach.** mobvibe does not drive vanilla Claude Code at all. It
launches `claude-code-acp` (Zed's adapter) and speaks the Agent Client
Protocol over the child's stdio. The daemon (Bun) wraps that stdio with
`ndJsonStream` and exchanges `initialize / newSession / prompt / cancel`
ACP-RPC frames. The CLI↔gateway transport is **Socket.io** (not raw
WebSocket), split across two namespaces `/cli` and `/webui`.

**Wire protocol.** Heavy and well-typed. RPC envelope is
`{ requestId, params, result?, error? }`. Persistent stream from daemon
to webui is `session:event` carrying a WAL-persisted
`SessionEvent { sessionId, machineId, revision, seq, kind, payload }`
with kinds like `user_message`, `agent_message_chunk`,
`agent_thought_chunk`, `tool_call`, `tool_call_update`,
`permission_request`, `permission_result`, `terminal_output`,
`session_info_update`, `session_error`, `usage_update`.

**Auth model.** Hybrid. Better Auth (email/password) on webui; Ed25519
device keypair on the daemon. Per-session DEK is generated by the CLI,
wrapped with a content keypair derived from the same master secret,
sent to the webui on subscribe — so the gateway sees encrypted blobs,
not plaintext.

**Tech stack.** Bun daemon (Socket.io-client, ACP SDK, tweetnacl);
Node/Express gateway (Socket.io, better-auth, drizzle + Postgres,
ioredis); React 19 + Vite 8 + Tailwind 4 SPA, plus a Tauri v2 wrapper
for the mobile/desktop shells.

**Subprocess invocation.** From the ACP registry, not hard-coded:
`npx -y @zed-industries/claude-code-acp`. No `--bare`, no
`--output-format stream-json`, no PTY. Confirmed via grep.

**What we can learn:**

- WAL-persisted event stream with `revision`/`seq` is a sound idea.
  Socket.io namespaces per role (`/cli`, `/webui`) cleanly separate
  concerns.
- Ed25519 device keypair + per-session DEK is a much better trust
  model than password cookies.
- ACP is becoming an industry-standard agent protocol (Zed,
  JetBrains, etc. adopting). mobvibe's bet on ACP gives it multi-agent
  support for free.

**Why we don't adopt it now.** Depends on `claude-code-acp` (a
third-party adapter maintained by Zed). That adapter was written for
Anthropic's API, so it's unclear whether it transparently forwards the
3P provider's auth when `ANTHROPIC_BASE_URL` is set. Adopting ACP
without that confirmation risks breaking the "works with any provider"
goal. **Re-evaluate later** if `claude-code-acp` confirms 3P
compatibility, or if Anthropic ships a first-party ACP adapter.

### 2.2 pocket-claude — (b) stream-json subprocess

> Reference implementation for the approach we adopted.
> — [zhdzh12138/pocket-claude](https://github.com/zhdzh12138/pocket-claude)

**Approach.** Subprocess. Launches `claude --bare --output-format
stream-json` per session; pipes NDJSON events on stdout back to the
browser over WebSocket. No PTY, no Anthropic bridge, no third-party
adapter.

**Wire protocol (adopted).** Simple unidirectional + a few control
frames:

```ts
// Client → Server
type WsClientMessage =
  | { type: 'send'; sessionId; text; projectPath? }
  | { type: 'permission_response'; sessionId; approved }
  | { type: 'attach'; sessionId }
  | { type: 'detach'; sessionId };

// Server → Client
type WsServerMessage =
  | { type: 'text'; sessionId; text }
  | { type: 'thinking'; sessionId; text }
  | { type: 'tool_use'; sessionId; tool; input }
  | { type: 'tool_result'; sessionId; output }
  | { type: 'permission_request'; sessionId; tool; input }
  | { type: 'done'; sessionId; cost?; duration_ms? }
  | { type: 'error'; sessionId; message };
```

The server translates stream-json's `assistant.message.content[]`
blocks into these simplified shapes. The UI doesn't need to know about
stream-json at all.

**Auth model.** None beyond HTTP (typical local SPA on loopback).

**Subprocess invocation.**

```
claude --bare --output-format stream-json
```

(Permission handling likely `--permission-mode bypassPermissions` for
v0.1 — we adopt the same. See architecture.md §9.2 for the v0.2 plan.)

**What we adopt:**

- Subprocess invocation shape.
- Wire protocol shape (`WsClientMessage` / `WsServerMessage`).
- Server-as-translator: stream-json in, simplified WS frames out.
- Multi-turn via stdin.

**What we extend:**

- Local hub mode (not just remote one-process server).
- Optionally dial out to a public hub (the original goal).
- Permission model upgrade in v0.2.

### 2.3 ly0/cc-remote-control-server — (a) Bridge replication

> "Self hosted Claude code Remote Control server"
> — [ly0/cc-remote-control-server](https://github.com/ly0/cc-remote-control-server)

**Approach.** Replicates Anthropic's own bridge protocol. The server
returns a "work secret" to the CLI; the CLI opens a session-ingress
WebSocket back to that secret's `api_base_url`; the server fakes
`/api/oauth/profile` so the CLI's profile check passes; the CLI is
patched at install time to swap `BASE_API_URL` from
`https://api.anthropic.com` to the user's server URL and to neutralize
the `tengu_ccr_bridge` feature flag.

**Wire protocol.** The CLI's own private bridge protocol
(`control_request` / `control_response` envelopes with subtypes
`initialize`, `set_permission_mode`, `interrupt`, `can_use_tool`).
Newer code path ("CCR v2") uses `cse_<hex>` session IDs, unsigned JWT
worker tokens, and SSE transport instead of WebSocket.

**Auth model.** None at the application layer. The `environment.secret`
returned on registration is never checked anywhere. The CLI is told any
random string works for `CLAUDE_CODE_OAUTH_TOKEN`.

**Self-hosting.** `npm install && npm start`. No Docker, no binary
release. In-memory state only — restart loses everything.

**What we can learn:**

- `can_use_tool` is a real `control_request` subtype that the CLI
  emits — useful confirmation that permission pauses can flow through
  the structured protocol when in bridge mode.
- The patch-script approach is the brittle way to go. ly0 hard-fails
  on non-CLI-version 2.1.112.

**Why we don't adopt it.** Patches the CLI's bundled JS to redirect
`frame.claudeusercontent.com`. Couples to a specific CLI version. The
CLI's bridge protocol is Anthropic-private and undocumented. **Fails
the provider-agnostic goal** because it specifically impersonates
Anthropic's infrastructure.

### 2.4 permissionnine9/claude-code-remote-control — (d) PTY + xterm

> "在移动端、网页端可远程控制你电脑本地的claude code" (Remotely
> control local claude code from mobile/web)
> — [permissionnine9/claude-code-remote-control](https://github.com/permissionnine9/claude-code-remote-control)

**Approach.** Launches `claude` inside a pseudo-terminal via
`@lydell/node-pty`, treats it as a TTY. Every PTY chunk is broadcast
raw to the browser, which renders it with xterm.js. A regex-based
parser also watches for `[y/n]` permission prompts and synthesizes
`permission_request` events.

**Wire protocol.** PTY bytes + a structured mirror:

```ts
// Client → Server (relevant subset)
| { type: 'send_message'; content; uuid? }
| { type: 'permission_response'; tool_use_id; behavior: 'allow'|'deny'; updated_input?; feedback? }
| { type: 'interrupt' }     // writes \x03 to PTY
| { type: 'input'; data }    // raw keystroke passthrough
| { type: 'resize'; cols; rows }
```

```ts
// Server → Client
| { type: 'connected'; sessionId }
| { type: 'stream'; event }                  // structured, largely unused
| { type: 'raw_output'; data }               // TTY bytes — what xterm paints
| { type: 'permission_request'; ... }        // synthesized from regex
| { type: 'status'; status; detail? }
```

Permission responses are implemented by literally writing `y\r` or
`n\r` to the PTY — the TUI's prompt accepts it. `updated_input` /
`feedback` fields are accepted on the wire but discarded; the TUI
doesn't understand them.

**Auth model.** Single shared bearer token printed in the banner and
embedded in a QR code. Optional `cloudflared tunnel` for public access.
No accounts, no Trusted Device, no rotation other than restart.

**Subprocess invocation.** Plain `claude`. The bridge passes
`--dangerously-skip-permissions` only when the user selects one of
`accept-edits` / `bypass-permissions` modes; in `default` mode the
bridge actually does pause on prompts.

**What we can learn:**

- **Negative lesson:** relying on TUI parsing for permissions is
  fragile. The `tool_use_id` correlation is fake (`perm-${Date.now()}`)
  because the PTY doesn't contain real IDs. Any Claude Code TUI
  redesign silently breaks permission detection.
- **Negative lesson:** rendering TUI bytes in xterm.js is bad UX for
  AI chat — you see the spinner, the syntax-highlighted JSON, the
  permission prompt. None of that is the conversation the user cares
  about.
- **Positive lesson:** a single bearer token + cloudflared tunnel is
  the minimum viable auth story. Worth a reference for our hub
  mode's first cut.

**Why we don't adopt it.** The UI is a terminal mirror, not a chat
UI. Permissions work only by accident of regex matching the TUI's
exact prompt text.

### 2.5 barjakuzu/claude-rc-launcher — (a') `/remote-control` handoff

> "A lightweight web UI to launch and manage Claude Code remote-control
> sessions from any browser"
> — [barjakuzu/claude-rc-launcher](https://github.com/barjakuzu/claude-rc-launcher)

**Approach.** Launches `claude` inside a detached tmux session. Waits
for the `❯` prompt, types `/remote-control`, waits for the literal
status string `"Remote Control active"`, parses the URL via regex
`https://claude\.ai/code/session_[^\s]+`, and hands the user off to
the official claude.ai/code web UI.

**Wire protocol.** None of its own. HTTP/1.1 JSON over Python's
stdlib `http.server.ThreadingHTTPServer`. All routes are stateless
JSON dicts; polling is the only "live" mechanism (700 ms pane refresh
in the worst case, 4–5 s elsewhere).

**Auth model.** Single shared password (`RC_AUTH_USER` /
`RC_AUTH_PASS`), CSRF-protected login form, HTTP session cookie
`rc_session`. Optional Cloudflare tunnel.

**Tech stack.** Backend: Python stdlib only — no FastAPI, no
`requests`, no DB. Frontend: React 18 + Vite + TypeScript (committed
build), with a vanilla `/legacy` fallback. `xterm.js` is used only as
a renderer for the polled tmux pane preview, not as a real-time
stream.

**Subprocess invocation.** Three launch modes via `RC_FLAGS`:

```python
"c":     "--dangerously-skip-permissions --verbose",
"ci":    "--dangerously-skip-permissions --teammate-mode in-process --verbose",
"safe":  "--verbose",
```

All run inside `tmux new-session -d`. None use `--bare`,
`--output-format stream-json`, or `--print`.

**What we can learn:**

- **Positive lesson:** state machines driven by `tmux capture-pane`
  + regex work, but only for the things Claude's TUI literally shows.
- **Positive lesson:** stdlib-only Python backend + committed JS
  bundle = "Node not required on target" — same deployability goal we
  have for the Bun binary.
- **Negative lesson:** once `/remote-control` is active, the launcher
  just hands off. The chat UI is claude.ai's. We don't get a self-
  hosted UI this way.

**Why we don't adopt it.** The chat UI is `claude.ai`, which requires
Anthropic auth. Doesn't satisfy the goal of providing a self-hostable
RemoteControl capability for non-Anthropic providers.

### 2.6 Zun-RZ/claude-remote-coding — (a') `/remote-control` plugin

> "Open background Claude Code Remote Control sessions per project,
> drivable from the Claude mobile app / claude.ai/code"
> — [Zun-RZ/claude-remote-coding](https://github.com/Zun-RZ/claude-remote-coding)

**Approach.** A Claude Code *plugin* (POSIX shell + PowerShell + one
Python file). Wraps the official `claude --remote-control` flag —
nothing more. Launches it inside tmux on Unix, ConPTY on Windows.
Provides a keystroke-inbox helper so that `/`-prefixed commands
(`/clear`, `/compact`, `/model haiku`) can be sent from outside the
TUI.

**Wire protocol.** None. The "protocol" is "newline-terminated line in
an inbox file = one keystroke sequence sent into the TUI". No
WebSocket, no JSON-RPC, no socket at all. Authentication is whatever
the official `--remote-control` flow does inside Anthropic.

**What we can learn:**

- **Negative lesson:** a plugin cannot escape Anthropic's bridge.
  All "drive from claude.ai/code" capability comes from Anthropic's
  binary.
- **Positive lesson:** a file-based inbox is a clean way to inject
  keystrokes into a TUI. Useful pattern if we ever need to support
  `--bare=false` mode (which keeps the TUI alive) — though our v0.1
  uses `--bare` and doesn't need it.

**Why we don't adopt it.** It's a wrapper around Anthropic's own
bridge. Fails the provider-agnostic goal entirely.

## 3. Comparative analysis

### 3.1 Provider agnosticism

| Approach | Provider agnostic? | Why |
| -------- | :----------------: | --- |
| (a) ly0 | No | Patches CLI to impersonate Anthropic infra |
| (a') launcher | No | Hands off to claude.ai; needs Anthropic OAuth |
| (b) pocket-claude | **Yes** | Subprocess uses whatever credential the CLI uses |
| (c) mobvibe | Unverified | Depends on `claude-code-acp`'s 3P behavior |
| (d) permissionnine9 | Yes | TUI captures whatever Claude outputs |

Only **(b)** is provably provider agnostic. **(d)** is also agnostic in
principle but with terrible UX. **(c)** is the most promising future
direction but requires confirmation.

### 3.2 Independence from Anthropic infra

| Approach | Independent? | Notes |
| -------- | :-----------: | ----- |
| (a) ly0 | No | Requires patch script targeting CLI version 2.1.112 |

| (c) mobvibe | Yes (E2EE) | Hosted gateway in metadata path |

## 4. Decision

**Adopt (b) — `claude --bare --output-format stream-json` subprocess,
pocket-claude style.**

> **Update (2026-07-02):** this "adopt a subprocess" decision was
> ultimately reversed. Helpers that launched subprocesses (`attach-orc`
> starting `claude`; `attach-tmux` starting `tmux` to mirror a pane)
> were built and then removed the same day — launching subprocesses is
> out of scope; open-rc ships a pure relay and the user brings their
> own bridge. The auth note still stands for any future subprocess
> launcher: use `--print`, not `--bare` (bare authenticates only via
> `ANTHROPIC_API_KEY`, so OAuth-login machines get "Not logged in").
> See [`architecture.md`](./architecture.md) §8.2.

Concrete reasons:

1. **Only approach provably satisfying the provider-agnostic goal.**
   ly0/launcher/Zun-RZ fail on first contact with Deepseek/GLM/MiniMax.
2. **Independent of Anthropic services.** No OAuth, no bridge, no
   patch script, no Trusted Device enrollment, no version pinning.
3. **Robust protocol.** stream-json is the public Agent SDK format.
   Documented. Stable. Tested by Anthropic themselves.
4. **Right complexity tier.** Smaller than mobvibe, larger than
   permissionnine9 — the surface area we actually need to control.
5. **Pattern already proven.** pocket-claude has shipped multi-turn
   sessions with this exact wire format.

**Reserve (c) ACP for the future.** If `claude-code-acp` confirms 3P
provider compatibility (or Anthropic ships a first-party adapter), the
ACP approach buys us multi-agent support for free. Not now.

**Reject (a), (a'), (d).** Documented above.

### Pure-relay pivot (Phase 7 addendum)

Earlier versions of `open-rc` shipped a server that launched one
`claude` subprocess per session (`open-rc attach` / `attach-orc`,
plus a server-side `SessionManager`). That model has been removed in
favor of a pure WebSocket relay:

- **`open-rc serve` starts no processes.** No `fork`, no `exec`, no
  process-creation calls of any kind. The server's process table is
  unchanged after `serve` boots.
- **`open-rc serve` does not manage anything.** No session lifecycle,
  no disk persistence, no `sessions.json` (only a bounded in-memory
  replay buffer per connected client, dropped on disconnect).
- **The user runs `claude` themselves.** Whatever pipeline they use
  to feed `stream-json` over a WebSocket to the relay is their
  problem — `open-rc` ships no bridge, by design.

Why this pivot:

- Take-over (terminate an external `claude` and replace it with an
  open-rc-managed subprocess) is permanently banned. Take-over
  requires the server to *find* `claude` processes, which requires
  walking `ps`/`lsof`/`/proc`. The cleanest way to keep that
  temptation off the table is to ship a server that cannot start
  processes at all.
- The relay model makes multi-host trivial: `claude` lives on the
  machine that has its working directory and credentials; the
  user's bridge lives there; the server can live anywhere reachable;
  the browser can live anywhere reachable.
- Provider agnosticism is preserved because the user still runs
  `claude --bare` themselves, with whatever `ANTHROPIC_BASE_URL`
  setup they already use.

We still don't chase `barjakuzu`'s `/remote-control` handoff (it needs
kill-and-swap, which we banned) or `permissionnine9`'s PTY/TTY
mirroring.

> **Update (2026-07-02):** a `attach-tmux` command that mirrored an
> existing tmux `claude` via `capture-pane`/`send-keys` (relaying the
> screen as `screen` frames) was briefly built and then removed the
> same day — it started `tmux`, and launching processes is out of
> scope. open-rc ships a pure relay; the user brings their own bridge.
> The server never touches a terminal.

## 5. What we learned that informs our design

- **The wire protocol can be tiny.** pocket-claude's seven message
  types are enough. We adopt that shape.
- **Permission pauses are the hardest part.** permissionnine9 shows
  the only way to get real pauses in `--bare` mode is... not possible.
  The CLI just doesn't emit permission events on stream-json. The
  options are: (i) accept bypassPermissions for v0.1, (ii) inject a
  PreToolUse hook via the subprocess's `--settings`, (iii) give up
  `--bare` and parse TUI prompts.
- **A self-signed QR-style pairing flow is the minimum viable hub
  auth.** permissionnine9 and barjakuzu both use it. We can do better
  with Ed25519 (mobvibe's pattern) but the surface is small.
- **WAL persistence with revision/seq ordering** (mobvibe's
  `SessionEvent`) is the right shape for replaying session history to
  a newly-attached client. Worth borrowing for hub mode.
